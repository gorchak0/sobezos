  "question": "3.15. Что делает оператор MERGE? Какие у него есть ограничения?",
  "answer": "**MERGE**\n\n`MERGE` (часто называют UPSERT или `ON DUPLICATE KEY UPDATE` в некоторых СУБД) позволяет выполнить **вставку или обновление записи** в таблице в зависимости от того, существует ли она уже. Это объединяет логику `INSERT` и `UPDATE` в одном операторе, что упрощает работу с дубликатами и синхронизацией данных.\n\n**Принцип работы:**\n\n1. **INSERT** — если запись с заданным ключом отсутствует, создаётся новая.\n\n2. **UPDATE** — если запись с ключом уже существует, она обновляется новыми значениями.\n\nПример (синтаксис для SQL Server / Oracle):\n\n`MERGE INTO employees AS target`\n\n`USING (SELECT 1 AS employee_id, 'John Doe' AS name) AS source`\n\n`ON target.employee_id = source.employee_id`\n\n`WHEN MATCHED THEN`\n\n`UPDATE SET name = source.name`\n\n`WHEN NOT MATCHED THEN`\n\n`INSERT (employee_id, name) VALUES (source.employee_id, source.name);`\n\n**Практическое использование:**\n\n- Синхронизация данных между таблицами или источниками.\n\n- Упрощение обработки «добавить или обновить».\n\n- Эффективная работа с ETL и массовыми обновлениями данных.\n\n**Ограничения и нюансы:**\n\n1. **Не является стандартным SQL** — синтаксис и поддержка зависят от СУБД.\n\n- MySQL: `INSERT ... ON DUPLICATE KEY UPDATE`\n\n- PostgreSQL: `INSERT ... ON CONFLICT DO UPDATE`\n\n2. **Производительность** — массовые MERGE могут быть медленными на больших таблицах без правильных индексов.\n\n3. **Триггеры и ограничения** — поведение может отличаться, особенно с внешними ключами и ограничениями уникальности.\n\n4. **Одновременные операции** — при высокой конкуренции нужно быть осторожным с блокировками и гонками данных."
}

  "answer": "**Агрегатные функции SQL**\n\nАгрегатные функции позволяют выполнять вычисления над набором строк и получать одно итоговое значение для группы. Они часто используются вместе с `GROUP BY` для анализа данных.\n\n**Основные функции:**\n\n- `COUNT()` — считает количество строк в наборе результатов. Можно использовать `COUNT(DISTINCT column)` для подсчёта уникальных значений.\n\n- `SUM()` — суммирует числовые значения столбца.\n\n- `AVG()` — вычисляет среднее значение числового столбца.\n\n- `MIN()` / `MAX()` — возвращают минимальное и максимальное значение соответственно.\n\n**Работа со строками:**\n\n- `GROUP_CONCAT()` / `STRING_AGG()` — объединяет строки в одну, используя разделитель. Отлично подходит для создания списков в одной колонке.\n\n**Статистические функции:**\n\n- `STDDEV()` / `STDEV()` — стандартное отклонение.\n\n- `VARIANCE()` — дисперсия.\n\n- `CORR()` — коэффициент корреляции между двумя числовыми столбцами.\n\n- `COVAR_POP()` / `COVAR_SAMP()` — популяционная и выборочная ковариация.\n\n**Аналитические функции (window functions):**\n\n- `FIRST()` / `LAST()` — первое и последнее значение в группе.\n\n- `PERCENTILE_CONT()` / `PERCENTILE_DISC()` — вычисляют перцентили.\n\n- `CUME_DIST()` — накопительное распределение значений.\n\n- `RANK()` / `DENSE_RANK()` — ранжируют значения в упорядоченном наборе данных.\n\n**Агрегация в массивы и JSON:**\n\n- `ARRAY_AGG()` / `JSON_AGG()` — собирают значения в массив или JSON-объект. Полезно для построения вложенных структур данных при аналитике или API.\n\n**Практические советы на интервью:**\n\n1. Многие функции доступны не во всех СУБД, поэтому важно знать аналоги для PostgreSQL, MySQL, SQL Server и Oracle.\n\n2. Агрегатные функции можно комбинировать с `GROUP BY` и фильтровать с `HAVING`.\n\n3. Window-функции (`RANK`, `CUME_DIST`) удобны для аналитики без потери исходных строк.\n\n4. Для больших таблиц стоит обращать внимание на индексы и группировки — некоторые агрегаты (особенно `DISTINCT`, `GROUP_CONCAT`) могут быть ресурсоемкими."

  "question": "3.17. Что такое ограничения (constraints)? Какие вы знаете?",
  "answer": "Ограничения (constraints) в SQL \\- это правила, устанавливаемые на столбцы таблицы, которые определяют, какие значения могут быть вставлены, обновлены или удалены в таблице.\n\nОсновные типы ограничений:\n\n1. **Primary Key** — определяет уникальный идентификатор строки. Нельзя вставить дубликат и нельзя оставить `NULL`.\n\n2. **Unique** — похож на первичный ключ, но может применяться к нескольким колонкам, и в отличие от PK может допускать `NULL`.\n\n3. **Foreign Key** — обеспечивает ссылочную целостность между таблицами. Например, нельзя вставить заказ на несуществующего пользователя. Также можно задать поведение при удалении или обновлении (`CASCADE`, `SET NULL` и т. д.).\n\n4. **Check** — кастомные условия на значения в колонке, например, чтобы возраст был больше 0 или дата окончания договора не раньше даты начала.\n\n5. **Default** — задаёт значение по умолчанию для колонки, если при вставке оно не указано.\n\n6. **Ограничения на удаление/обновление (ON DELETE / ON UPDATE)** — определяют поведение при изменении связанных записей: каскадное удаление, установка `NULL`, запрет и т. д.\n\nНа практике я стараюсь использовать ограничения максимально, потому что они убирают часть бизнес-логики из приложения и гарантируют защиту на уровне данных. Даже если где\\-то в коде закралась ошибка, constraint не даст положить мусор в базу."

  "question": "3.18. Что такое суррогатные ключи?",
  "answer": "Суррогатный ключ — это искусственный первичный ключ, который не несёт бизнес-смысла и существует только для обеспечения уникальности строки. Чаще всего это авто_инкрементируемое число или UUID, которое генерируется базой при вставке записи.\n\nОсновные моменты:\n\n- **Уникальность:** каждый суррогатный ключ гарантированно уникален.\n\n- **Независимость от данных:** он не зависит от бизнес-атрибутов и не меняется при изменении данных.\n\n- **Производительность:** сравнивать и индексировать целочисленный ключ проще и быстрее, чем, например, составной натуральный ключ из нескольких колонок.\n\n- **Прозрачность:** для пользователей и бизнес-логики этот ключ, как правило, скрыт. Никто не будет опираться на `id` в реальном бизнес-контексте, он нужен только для связей и внутренней целостности.\n\nКлассический пример — поле `id SERIAL` в PostgreSQL или `AUTO_INCREMENT` в MySQL.\n\nОбычно суррогатные ключи используют вместе с естественными ключами (например, email пользователя или артикул товара). Естественный ключ полезен для бизнес-логики, а суррогатный — для ссылочной целостности и стабильности схемы. Такой подход упрощает жизнь при рефакторинге: бизнес-атрибуты могут измениться, а суррогатный ключ останется тем же самым."
}
